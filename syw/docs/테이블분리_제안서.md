# STORAGE_ITEM 테이블 분리 제안서

## 1. 현황

현재 파일과 폴더를 `STORAGE_ITEM` 단일 테이블에서 `item_type` (FILE/DIRECTORY)으로 구분하고 있다.

```sql
CREATE TABLE storage_item (
    id             BIGINT AUTO_INCREMENT PRIMARY KEY,
    parent_id      BIGINT,
    owner_id       BIGINT,
    display_name   VARCHAR(255),
    stored_name    VARCHAR(36),     -- 폴더는 NULL
    size           BIGINT,          -- 폴더는 NULL
    item_type      VARCHAR(50),     -- FILE or DIRECTORY
    content_type   VARCHAR(255),    -- 폴더는 NULL
    extra_metadata TEXT,            -- 폴더는 NULL
    ...
);
```

**문제**: 폴더 행은 `stored_name`, `size`, `content_type`, `extra_metadata`가 전부 NULL.

---

## 2. 제안: FOLDER / FILE 테이블 분리

```sql
CREATE TABLE folder (
    id               BIGINT AUTO_INCREMENT PRIMARY KEY,
    owner_id         BIGINT NOT NULL,
    display_name     VARCHAR(255) NOT NULL,
    parent_folder_id BIGINT,              -- FK → folder.id (Self-reference)
    created_at       DATETIME,
    updated_at       DATETIME,
    deleted_at       DATETIME,
    INDEX idx_parent (parent_folder_id, deleted_at),
    INDEX idx_owner_parent (owner_id, parent_folder_id, display_name)
);

CREATE TABLE file (
    id             BIGINT AUTO_INCREMENT PRIMARY KEY,
    owner_id       BIGINT NOT NULL,
    display_name   VARCHAR(255) NOT NULL,
    folder_id      BIGINT,                -- FK → folder.id
    stored_name    VARCHAR(36) NOT NULL,
    size           BIGINT NOT NULL,
    content_type   VARCHAR(255),
    extra_metadata TEXT,
    created_at     DATETIME,
    updated_at     DATETIME,
    deleted_at     DATETIME,
    INDEX idx_folder (folder_id, deleted_at),
    INDEX idx_owner_folder (owner_id, folder_id, display_name),
    INDEX idx_stored_name (stored_name)
);
```

## 3. 왜 분리해야 하는가

### 3.1 트리 연산은 폴더끼리만 일어난다

파일 시스템의 핵심 연산을 분석하면, **트리 구조 탐색은 폴더 간에만 발생**한다.

| 연산 | 탐색 대상 | 파일 관여 |
|------|----------|----------|
| 재귀 삭제 (BFS) | FOLDER → FOLDER → FOLDER | 각 폴더의 FILE 수집 (별도) |
| 순환 참조 검증 | FOLDER → FOLDER → FOLDER (조상 체인) | 없음 |
| 폴더 이동 | FOLDER의 parent_id 변경 | 없음 |
| 폴더 복사 | FOLDER 트리 복제 → FILE 복제 | 별도 단계 |

단일 테이블에서 트리를 순회하면, **인덱스에 파일 행이 섞여있어** 불필요한 데이터를 읽게 된다.

### 3.2 규모별 성능 영향

파일:폴더 비율을 일반적인 50:1로 가정:

| 규모 | 파일 수 | 폴더 수 | 단일 테이블 인덱스 | 분리 시 폴더 테이블 | 트리 순회 차이 |
|------|---------|---------|-------------------|-------------------|--------------|
| 소규모 | 1만 | 1천 | 11,000행 | 1,000행 | 없음 |
| 중규모 | 100만 | 1만 | 1,010,000행 | 10,000행 | 미미 |
| **대규모** | **1,000만** | **10만** | **10,100,000행** | **100,000행** | **체감 시작** |
| 초대규모 | 1억+ | 100만+ | 1억+행 | 100만행 | **명확한 차이** |

**대규모(파일 1,000만+) 이상**에서 분리의 이점이 본격적으로 나타난다.

이유:
- 분리된 폴더 테이블(10만 행)은 **전체가 메모리에 올라감** → 트리 탐색이 거의 메모리 연산
- 단일 테이블(1,000만+ 행)은 인덱스 자체가 커서 디스크 I/O 발생 가능
- 트리 순회 시 `item_type = 'DIRECTORY'` 필터링 비용이 행 수에 비례하여 증가

### 3.3 Aggregate 경계의 명확화

DDD 관점에서 파일과 폴더는 **책임이 다른 엔티티**다.

| 관점 | FOLDER | FILE |
|------|--------|------|
| 역할 | 컨테이너 (트리 노드) | 데이터 보유자 |
| 물리 저장 | 없음 | UUID 파일 존재 |
| 고유 속성 | parent_folder_id (자기참조) | stored_name, size, content_type |
| NULL 컬럼 | 없음 | 없음 |

단일 테이블에서는 폴더 행이 파일 전용 컬럼(stored_name, size, content_type, extra_metadata)을 **전부 NULL로 보유**한다. 분리하면 각 테이블이 자기 컬럼만 가지므로 NOT NULL 제약을 걸 수 있고, 데이터 정합성이 높아진다.

## 4. 트레이드오프

### 분리의 비용

| 항목 | 영향 |
|------|------|
| **폴더 내 목록 조회** | 단일 테이블: `WHERE parent_id = ?` 1번 → 분리: FOLDER + FILE 각각 조회 (2번 or UNION) |
| **코드 복잡도** | 엔티티 2개, 리포지토리 2개, 서비스 로직 분기 |
| **리팩토링 비용** | 기존 Step 1~2 코드 전면 수정 |
| **조인 비용** | 파일의 폴더 경로를 알려면 folder 테이블 조인 필요 |

### 분리의 이점

| 항목 | 영향 |
|------|------|
| **트리 순회 성능** | 폴더 테이블만 탐색, 파일 수에 무관 |
| **NULL 제거** | 모든 컬럼에 NOT NULL 가능, 데이터 정합성 향상 |
| **인덱스 효율** | 각 테이블에 최적화된 인덱스 설계 가능 |
| **Aggregate 명확화** | 폴더와 파일의 도메인 책임 분리 |
| **독립적 확장** | 파일에 버전 관리, 폴더에 공유 권한 등 독립 확장 용이 |

## 5. 폴더 목록 조회 성능 대응

분리 시 가장 큰 비용인 "폴더 내 목록 조회 2번 쿼리" 문제의 대응 방안:

### 방안 A: 2번 쿼리 허용

```java
List<Folder> subFolders = folderRepository.findByParentFolderIdAndDeletedAtIsNull(folderId);
List<File> files = fileRepository.findByFolderIdAndDeletedAtIsNull(folderId);
// 합쳐서 반환
```

- 각 쿼리가 인덱스 최적화되어 있으므로 개별 성능은 우수
- 네트워크 왕복 1회 추가 (DB 커넥션 내에서 2번 쿼리)

### 방안 B: UNION 쿼리

```sql
SELECT id, display_name, 'DIRECTORY' as item_type, NULL as size, created_at
FROM folder WHERE parent_folder_id = ? AND deleted_at IS NULL
UNION ALL
SELECT id, display_name, 'FILE' as item_type, size, created_at
FROM file WHERE folder_id = ? AND deleted_at IS NULL
ORDER BY item_type, display_name;
```

- 1번 쿼리로 해결
- 단, 컬럼 구조를 맞춰야 하므로 NULL 패딩 필요

### 방안 C: DB View 활용

```sql
CREATE VIEW storage_item_view AS
SELECT id, owner_id, display_name, parent_folder_id as parent_id,
       'DIRECTORY' as item_type, NULL as stored_name, NULL as size, ...
FROM folder
UNION ALL
SELECT id, owner_id, display_name, folder_id as parent_id,
       'FILE' as item_type, stored_name, size, ...
FROM file;
```

- 조회는 기존 단일 테이블과 동일한 인터페이스 유지
- 쓰기는 각 테이블에 직접 수행

## 6. 마이그레이션 계획

### Phase 1: 현행 유지 + Step 3 구현 완료
- 단일 테이블로 Step 3 (폴더 삭제/이동/복사) 구현
- 기능 정확성 검증 완료

### Phase 2: 성능 테스트
- 대량 데이터 삽입 (파일 100만+, 폴더 1만+)
- 재귀 삭제, 폴더 이동, 목록 조회 성능 측정
- 단일 테이블의 병목 지점 확인

### Phase 3: 테이블 분리 (필요 시)
- `folder`, `file` 테이블 생성
- 기존 `storage_item` 데이터 마이그레이션
- 서비스 레이어 리팩토링
- 성능 재측정 → 개선 확인
