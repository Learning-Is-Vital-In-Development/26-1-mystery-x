# 물리 파일 복사 제거 제안서

## 1. 현황

현재 파일 복사 시 **물리 파일도 함께 복사**하고 있다.

```java
// FileManager.copy
Path sourcePath = resolvePath(sourceStoredName);
Path targetPath = resolvePath(targetStoredName);
Files.copy(sourcePath, targetPath);
```

복사된 파일은 새로운 UUID를 부여받아 독립적인 물리 파일로 존재한다.

```
원본: stored_name = "aaa-111" → /uploads/aa/a1/aaa-111
복사: stored_name = "bbb-222" → /uploads/bb/b2/bbb-222  (동일 내용)
```

**문제**: 동일한 내용의 파일이 디스크에 중복 저장되어 공간 낭비가 발생한다.

---

## 2. 제안: 물리 파일 공유 + 참조 카운팅

복사 시 물리 파일을 복제하지 않고, **같은 stored_name을 참조**하는 DB 레코드만 생성한다.

```
원본 StorageItem (stored_name: "aaa-111") ──┐
                                             ├── 물리 파일 1개 공유
복사 StorageItem (stored_name: "aaa-111") ──┘
```

### 필요한 변경

#### 2.1 참조 카운트 관리

물리 파일 삭제 시, 해당 `stored_name`을 참조하는 레코드가 더 이상 없을 때만 삭제해야 한다.

```java
// 삭제 시
long refCount = repository.countByStoredNameAndDeletedAtIsNull(storedName);
if (refCount == 0) {
    fileManager.delete(storedName);
}
```

#### 2.2 복사 로직 단순화

```java
// 변경 전: 물리 파일 복사 + 새 UUID
String newStoredName = uuidGenerator.generate().toString();
fileManager.copy(source.getStoredName(), newStoredName);

// 변경 후: DB 레코드만 생성, 같은 stored_name 공유
StorageItem copy = new StorageItem(
        targetParentId, source.getOwnerId(), source.getDisplayName(),
        source.getStoredName(),  // 동일 stored_name 사용
        source.getSize(), ItemType.FILE,
        source.getContentType(), source.getExtraMetadata()
);
```

## 3. 트레이드오프

### 공유의 이점

| 항목 | 영향 |
|------|------|
| **디스크 절약** | 대용량 파일 복사 시 효과 큼 (예: 100MB 파일 10회 복사 → 100MB vs 1GB) |
| **복사 속도** | 물리 I/O 없이 DB INSERT만으로 즉시 완료 |
| **보상 트랜잭션 불필요** | 물리 파일 복사가 없으므로 cleanup 로직 제거 가능 |

### 공유의 비용

| 항목 | 영향 |
|------|------|
| **삭제 복잡도 증가** | 모든 삭제 시 참조 카운트 확인 필요 |
| **동시성 문제** | 동시 삭제 시 카운트 race condition 가능 → 비관적 락 또는 별도 ref_count 테이블 필요 |
| **파일 수정 불가** | 파일 내용 수정(덮어쓰기) 기능 추가 시 COW(Copy-on-Write) 필요 |
| **복구 복잡도** | soft delete 복구 시 물리 파일 존재 여부가 다른 레코드에 의존 |

---

## 4. 동시성 문제 상세

가장 큰 리스크는 **동시 삭제 시 참조 카운트 정합성**이다.

```
시점 1: A 삭제 요청 → refCount 조회 = 2 → 물리 파일 유지
시점 2: B 삭제 요청 → refCount 조회 = 1 → 물리 파일 삭제
시점 3: A soft delete 완료 → 실제 refCount = 0 (정상)

BUT 순서가 꼬이면:
시점 1: A 조회 refCount = 2
시점 2: B 조회 refCount = 2
시점 3: A 삭제 → refCount 1이라 유지
시점 4: B 삭제 → refCount 1이라 유지 → 물리 파일 미삭제 (orphan)
```

대응 방안:
- **방안 A**: `SELECT ... FOR UPDATE`로 비관적 락
- **방안 B**: 별도 `file_reference` 테이블로 참조 카운트 명시 관리
- **방안 C**: 배치 작업으로 orphan 물리 파일 주기적 정리

## 5. 결론

| 항목 | 판단 |
|------|------|
| 현재 시점 | 물리 파일 복사 유지 (현행) |
| 전환 시점 | 디스크 사용량이 문제되거나 대용량 파일 복사가 빈번할 때 |
| 전환 기준 | 복사된 파일의 디스크 점유율 측정 후 판단 |
| 리스크 | 삭제 시 참조 카운팅 + 동시성 처리 복잡도 증가 |

**현재는 물리 파일 복사 방식을 유지하고, 디스크 효율이 문제될 때 참조 카운팅 방식으로 전환하는 것을 제안한다.**
